Detailed Findings

C-01 — IDOR: Cross-Account URL Deletion via GET Request
CRITICAL

Endpoint: url-del.php?id=<hex_id>

Method: GET (no POST required)

Description: The URL deletion endpoint does not verify that the URL being deleted belongs to the authenticated user. Any authenticated user can delete any other user's exam URLs by simply visiting url-del.php?id=TARGET_URL_ID.

Proof of Concept
# Account 1 (xcsectest@proton.me) owns URL ID: 8dceb6
# Account 2 (xcsec2@proton.me) — different user

# Account 2 deletes Account 1's URL:
curl -b "PHPSESSID=[account2_session]" \
  "https://cbt.mimikridev.com/url-del.php?id=8dceb6"
# → 302 redirect to url.php (success)

# Verify: Account 1's url.php no longer shows URL 8dceb6 — DELETED

Impact: An attacker with any registered account can delete all exam URLs for every school using the platform, disrupting exams nationwide. Combined with CSRF (H-01), this can be done without authentication via a malicious <img src="https://cbt.mimikridev.com/url-del.php?id=TARGET"> tag.

CVSS: 9.1 (Critical)

C-02 — IDOR: Cross-Account URL Modification
CRITICAL

Endpoint: url-update.php?id=<hex_id>

Method: POST

Description: The URL update endpoint does not verify ownership. Any authenticated user can modify any other user's exam URLs — changing the exam name, link, timer, and display settings.

Proof of Concept
# Account 2 modifies Account 1's URL (id: a5c1fc):
curl -b "PHPSESSID=[account2_session]" \
  "https://cbt.mimikridev.com/url-update.php?id=a5c1fc" \
  -X POST \
  -d "url_name=HACKED_BY_IDOR&url_link=https://hacked.example.com&timer=999&update="
# → 302 redirect to url.php (success)

# Account 1's url.php now shows:
#   url_name: HACKED_BY_IDOR
#   url_link: https://hacked.example.com
#   timer: 999

Impact: An attacker can redirect any school's exam URLs to phishing pages. Students opening the "exam browser" would be sent to an attacker-controlled site. This enables credential theft, fake exam submission, and data harvesting at scale. The exam timer can also be manipulated.

CVSS: 9.1 (Critical)

H-01 — No CSRF Protection on Any Form
HIGH

Affected: All forms — settings update, password change, URL add, URL delete, URL update

Description: Zero CSRF tokens are used anywhere in the application. All forms use method="POST" action="" with no anti-CSRF mechanism. Combined with the IDOR vulnerabilities, this allows unauthenticated remote attackers to craft malicious pages that execute actions on behalf of logged-in users.

Form Structure (setting.php)
<form method="POST" action="">
  <!-- No CSRF token -->
  <input name="nama_sekolah" type="text" value="...">
  <input name="img_logo" type="text" value="...">
  <button type="submit" name="update">Simpan</button>
</form>

<form method="POST" action="">
  <!-- No CSRF token -->
  <input name="old_password" type="password">
  <input name="new_password" type="password">
  <button type="submit" name="change_password">Ubah</button>
</form>

CVSS: 8.1 (High)

H-02 — CSRF via GET on URL Delete
HIGH

Endpoint: url-del.php?id=<hex_id>

Description: URL deletion is performed via GET request with only a client-side JavaScript confirm() dialog as "protection". This means deletion can be triggered by:

  • <img src="https://cbt.mimikridev.com/url-del.php?id=TARGET">
  • Iframe, link click, CSS background-image
  • Any context that loads a URL automatically

CVSS: 7.5 (High)

H-03 — Broken Password Reset Flow
HIGH

Endpoint: reset-password.php → process.php

Description: The password reset page at reset-password.php displays a "change password" form without requiring a valid reset token. The form contains only password and cpassword fields — no hidden token field. The backend (process.php:789) attempts to read a token key but it's never provided by the form, causing a PHP warning.

Error Response
Warning: Undefined array key "token" in
/www/wwwroot/172.93.219.140/cbt.mimikridev.com/process.php on line 789

Impact: The password reset mechanism is broken by design. If the token validation on the backend is weak (e.g., comparing against null/empty), this could allow arbitrary password reset. At minimum, it exposes the server path and the presence of a token-based system.

CVSS: 7.3 (High)

CVSS: 5.3 (Medium)

M-02 — SQL Error Disclosure / Database Schema Leak
MEDIUM

Trigger: POST to setting.php without img_logo field

Fatal error: Uncaught mysqli_sql_exception:
Column 'img_logo' cannot be null in
/www/wwwroot/172.93.219.140/cbt.mimikridev.com/setting.php:52

Impact: Reveals database column names, NOT NULL constraints, and confirms MySQL + mysqli driver usage. Aids SQL injection attempts.

CVSS: 5.3 (Medium)

M-03 — PHP Debug/Error Display in Production
MEDIUM

Multiple PHP error types displayed to end users: Deprecated, Warning, Fatal error. Production should have display_errors = Off and error_reporting = 0.

CVSS: 5.3 (Medium)

M-04 — Missing HTTP Security Headers
MEDIUM

No security headers present in HTTP responses:

  • X-Frame-Options — missing → clickjacking possible
  • Content-Security-Policy — missing
  • X-Content-Type-Options — missing
  • Strict-Transport-Security — missing (despite HTTPS)
  • Referrer-Policy — missing
  • Permissions-Policy — missing

CVSS: 5.0 (Medium)

M-05 — Session Cookie Missing Security Flags
MEDIUM

Initial PHPSESSID cookie set without Secure, HttpOnly, or SameSite flags:

Set-Cookie: PHPSESSID=xxx; path=/

Note: Dashboard page later sets a more secure cookie with secure; HttpOnly; SameSite=Strict, but the initial session establishment lacks these flags, creating a window for session hijacking.

CVSS: 4.8 (Medium)

M-06 — Predictable URL IDs (6-char Hex)
MEDIUM

URL entries are identified by 6-character hexadecimal IDs (e.g., 26eb5b, a5c1fc, 8dceb6). This gives only ~16.7 million possible values — feasible to brute-force. Combined with the IDOR vulnerabilities (C-01, C-02), an attacker can enumerate and modify all URLs on the platform.

CVSS: 4.3 (Medium)

M-07 — Open Registration Without Email Verification
MEDIUM

The signup form at signup.php creates accounts immediately without email verification. Registration requires: nama_sekolah, email, password, cpassword. A random username is auto-generated (e.g., cbt929b7e).

Impact: Enables mass account creation for abuse (spam, IDOR exploitation, resource exhaustion).

CVSS: 4.3 (Medium)

Kill Chain

Attack Scenario: Mass Exam Hijack

Attacker registers account via open signup (no email verification)
Brute-force 6-char hex URL IDs (16.7M space) via url-update.php?id=XXXXXX
For each valid ID: IDOR update → replace exam URL with phishing page
Students open "Exam Browser Client" → redirected to attacker's fake exam
Harvest student credentials, exam answers, personal data
Alternative: IDOR delete → destroy all exam URLs → deny service to schools

Attack Scenario: CSRF Exam Destruction

Attacker knows target school's URL IDs (visible to any authenticated user)
Craft malicious page: <img src="https://cbt.mimikridev.com/url-del.php?id=TARGET">
Send link to school admin (social engineering)
Admin visits page → browser auto-loads img → URL deleted without consent
No CSRF token → no protection → exam URLs destroyed silently
